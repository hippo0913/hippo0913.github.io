## 核心原则

### 1. 明确目标，而非步骤

❌ "请打开 src/main.py，找到第 30 行，把那个函数改成..."
✅ "修复用户登录时的 500 错误"

> 💬 hippo：让 Claude 自己决定怎么做，你只负责定义"什么算完成"。

### 2. 小步迭代，而非一次到位

❌ "把整个项目重构为微服务架构"
✅ "先把用户认证模块抽离成独立服务，我们看看效果"

> 💬 hippo：复杂任务拆小了，Claude 不容易犯错，你也更容易把关。

### 3. 提供上下文，而非假设

❌ "修复这个 bug"（什么都不给）
✅ "用户报告登录时出现 500 错误，这是错误日志...这是相关代码..."

> 💬 hippo：Claude 不是算命先生。你给的信息越完整，它的判断越准确。

## 提示词技巧

### 角色设定

```
你是一个资深 Python 开发者，擅长 Django 框架和 RESTful API 设计。
请帮我审查这段代码，找出潜在问题。
```

### 任务分解

```
请先：
1. 读取 src/auth/login.py 和 tests/test_login.py
2. 分析登录流程
3. 找出可能导致 500 错误的原因
4. 提出修复方案
```

### 输出格式

```
请按以下格式输出：
- 问题分析（3-5 点）
- 修复方案（按优先级排序）
- 需要修改的文件列表
```

## 常见场景最佳实践

### 场景 1：新功能开发

```
1. 先让 Claude 读现有代码结构
2. 描述需求，让它设计 API 接口
3. 确认设计后再写代码
4. 写完立即跑测试
```

### 场景 2：Bug 修复

```
1. 提供错误日志和复现步骤
2. 让 Claude 定位问题
3. 审查修复方案
4. 修复后添加回归测试
```

### 场景 3：代码重构

```
1. 先让 Claude 分析代码异味（code smell）
2. 确定重构范围（不要一次重构整个项目）
3. 确保测试覆盖
4. 小步提交，每步验证
```

### 场景 4：学习新代码库

```
1. "帮我画一下这个项目的架构图"
2. "这个模块的主要职责是什么"
3. "这两个文件之间是什么关系"
4. "如果我要加 X 功能，应该改哪里"
```

## hippo 的注释

### 我最常用的三个技巧

> 💬 hippo：
> 1. **"先说计划"**——让它先讲怎么做，我再确认。这避免了大量返工。
> 2. **"扮演审查者"**——让它审查自己的代码，经常能发现它第一遍没注意到的问题。
> 3. **"为什么"**——问它为什么这么写，能帮我看懂它的思路。

### 我踩过的坑

> 💬 hippo：
> - 有次让它"优化整个项目的性能"，结果它改了一堆不该改的地方
> - 后来我学会了一次只让它优化一个函数，而且要先说清楚"什么算优化成功"
