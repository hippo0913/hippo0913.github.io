# 精读官方文档：Claude Code 如何工作

> 💬 hippo：这是系列的第五篇。理解 Claude Code 的工作原理，能帮你更好地使用它——知道它"怎么想"，才知道怎么"下指令"。

---

## 开篇：为什么要了解工作原理

很多人用 Claude Code，停留在"它很聪明"的层面。这不够。

**了解工作原理的价值**：
- 知道它什么时候可靠、什么时候可能犯错
- 理解为什么有些提示词有效、有些无效
- 学会预判它的行为，而不是事后纠正

---

## 核心概念：Claude Code 的工作流程

### 1. 读取上下文

Claude Code 启动时，会读取：
- 项目根目录的 `CLAUDE.md`
- 当前工作目录的文件结构
- 你的提示词和历史对话

> 💬 hippo：这就是为什么 CLAUDE.md 要保持简短——它在每次对话开始时都被读取，太长了会被"忽略"。

### 2. 理解任务

Claude 会分析你的请求：
- 这是什么类型的任务（读代码、写代码、调试、重构）
- 需要哪些工具（读文件、写文件、执行命令）
- 任务的依赖关系（先读再改，改完再测）

### 3. 规划执行

Claude 会生成一个执行计划：
1. 读取相关文件
2. 分析当前状态
3. 进行修改
4. 运行验证（测试、构建）
5. 根据结果调整

> 💬 hippo：这个过程是自动的，但你可以要求它"先说计划再执行"，这样你能提前发现问题。

### 4. 执行与反馈

Claude 执行每个步骤，并：
- 观察执行结果（命令输出、测试结果）
- 根据反馈调整策略
- 遇到错误时尝试修复

## 关键机制

### 上下文窗口

Claude 的上下文窗口是有限的（不同模型大小不同）：
- **Haiku 4.5**: 约 200K tokens
- **Sonnet 4.6**: 约 200K tokens
- **Opus 4.6**: 约 200K tokens

> 💬 hippo：200K tokens 约等于 15-20 万汉字。听起来很多，但如果 Claude 读取了整个代码库，很快就用完了。所以要用 `/compact` 或 `/clear` 来管理上下文。

### 工具调用

Claude Code 可以使用的工具：
| 工具 | 用途 |
|------|------|
| Read | 读取文件内容 |
| Write | 写入/修改文件 |
| Bash | 执行 shell 命令 |
| Glob | 搜索文件 |
| Grep | 搜索内容 |
| Agent | 启动子代理 |

### 子代理（Subagents）

当任务复杂时，Claude 可以启动子代理：
- 子代理有独立的上下文
- 主代理协调多个子代理
- 适合并行处理独立任务

> 💬 hippo：我给子代理的定义是"专职工人"——比如专门处理文档的 agent，专门跑测试的 agent。但不要用得太滥，信息隔离会让主代理失去全局视角。

## hippo 的注释

### 理解"代理"的本质

> 💬 hippo：Claude Code 不是"代码补全工具"，而是"代理编码工具"。这个区别很关键——它不只是帮你写完下半句，而是能理解整个任务并主动执行。

**我的使用心得**：
1. 给它明确的目标，而不是具体的步骤
2. 让它先说计划，你再确认
3. 复杂任务拆分成多个 session，避免上下文爆炸

### 上下文管理的艺术

> 💬 hippo：我踩过最大的坑就是"让它一直做"，结果上下文满了之后它开始胡言乱语。

**我的经验**：
- 完成一个大功能后，主动 `/clear` 重启
- 重启前让它写个进度总结到 `.md` 文件
- 新 session 读这个文件继续

---

## 常见误解

### ❌ "它记得之前 session 的一切"
✅ 每个 session 是独立的，除非你写了 CLAUDE.md 或进度文件

### ❌ "给它越多文件越好"
✅ 给它**相关**的文件，不是**所有**文件

### ❌ "它不会犯错"
✅ 它会犯错，而且犯错后会一本正经地坚持错误答案

---

## 一句话总结

Claude Code 的工作流程是：读取上下文 → 理解任务 → 规划执行 → 执行与反馈——理解这个流程，能帮你更好地下指令和管理预期。
